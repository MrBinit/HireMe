# HireMe Presentation Overview

- Candidate Career Page: "We built a public career flow where candidates can see active roles and submit one application with resume upload; role-window and pause/close checks prevent stale submissions."
- Job Opening Management: "Admins can create and manage structured job openings with role, team, location, requirements, and open/close windows; status controls include open, paused, and closed."
- Admin Login + Dashboard: "Admin access is JWT-protected; the dashboard gives filters, candidate pipeline visibility, resume download, and lifecycle tracking in one place."
- Candidate Detail + Manual Override: "Each candidate profile shows parse output, AI score, research summary, and status history; admins can override decisions with notes and trigger downstream actions."
- Backend API Layer (FastAPI): "The backend is async and split by clear REST endpoints for jobs, applications, admin actions, and integrations."
- Database (PostgreSQL): "Postgres is the source of truth for openings, applications, references, statuses, and audit-style progression fields."
- Resume Storage (S3): "We store resume binaries in S3 and keep metadata in Postgres, so storage and transactional data stay cleanly separated."
- Queue Architecture (SQS + Workers): "Heavy jobs run asynchronously through queues for parsing, evaluation, research, scheduling, and webhook side effects to keep APIs fast."
- Phase 02 Screening Stage 1 (Deterministic Filter): "Before LLM scoring, we run a low-cost rules filter on experience plus skills/keyword match to reduce cost and latency."
- Phase 02 Screening Stage 2 (LLM Scoring): "Qualified candidates go through rubric-based AI scoring (0–100) with structured JSON validation and threshold-driven shortlist/reject behavior."
- Research Enrichment: "Shortlisted candidates get LinkedIn/GitHub/portfolio enrichment, discrepancy checks, confidence signals, and a manager-ready 3–5 sentence brief."
- Referee Flow: "Referees are first-class users with dedicated login and reference submission; backend validation prevents references being attached to the wrong candidate."
- Phase 03 Scheduling Orchestration: "We generate real calendar-based slots, create hold events before sending options, and finalize one slot with race-condition protection."
- No-Reply + Reschedule Handling: "We send 24-hour reminders, auto-release unconfirmed holds at 48 hours, and support manager-approved reschedule loops."
- Phase 04 Interview Notetaker (Fireflies): "Interview transcript sync is integrated end to end; fallback behavior keeps pipeline continuity when transcript extraction is incomplete."
- Offer Letter Generation (5A): "Manager inputs are converted into AI-assisted offer drafts with human approval before sending."
- E-Signature Workflow (5B, DocuSign): "Offer PDFs are sent via DocuSign, webhook status updates drive state changes, and signed completion triggers immediate alerts."
- Slack Onboarding Trigger (Phase 06): "After signature completion, candidate onboarding is triggered in Slack with AI-personalized welcome messaging and HR notification."
- Security & Reliability Controls: "JWT auth, hashed passwords, webhook idempotency, and async durable side effects were implemented to improve production safety."
- CI/CD and Deployment: "GitHub Actions builds amd64 images, pushes to ECR, and deploys to EC2 over SSH with AWS Secrets Manager-backed runtime secrets."
- Edge-Case Hardening: "Key protections include duplicate-application prevention, upload validation, closed/paused-role blocking, slot-conflict prevention, and AI-confidence/manual-review gates."
- Trade-off Narrative: "We prioritized end-to-end reliability and delivery speed first, with clearly documented next-step hardening in transcript fidelity, AI evaluation, and infrastructure resilience."
